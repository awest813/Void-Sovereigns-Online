# Style Guide — Void Sovereigns Online

## Visual Identity

### Reference Points
- **Cowboy Bebop** — cool, worn, functional aesthetics; lived-in ships; jazz and silence
- **Ghost in the Shell (original)** — tactile technology, urban decay, quiet dread
- **NASA mission hardware** — matte finishes, utility labels, modular construction
- **70s–80s industrial design** — chunky panels, analog gauges, function over form

### Color Language

| Context | Direction |
|---------|-----------|
| Station interiors | Amber, grey, deep blue-grey. Dim lighting. Functional. |
| Ashwake Belt | Dark, cold greens and greys. Orange hazard accents. |
| Combat UI | Red/orange warnings. Minimal. Information-dense. |
| Lore / codex | Cream/sepia backgrounds. Clean serif typography. |
| Void/Psi elements | Deep violet, dark teal, occasional bioluminescent detail. Unsettling. |
| Positive/reward | Soft gold, warm white. Never garish. |

### Typography
- Display: wide, industrial, slightly worn — not futuristic chrome
- UI: clean mono or condensed sans-serif
- Lore text: readable serif, no decorative fonts
- All caps for section headers in-game; title case for NPC names

### Visual Hierarchy
- Prioritize readability over decoration
- Dense menus are acceptable if organized — this is an RPG, not a casual game
- No bloom, lens flare, or unnecessary particle effects on UI elements

---

## Audio Direction

- Ambient sound: station hum, distant machinery, air recyclers
- Music: quiet, jazz-influenced, understated — not epic orchestra
- Combat: percussive, punchy, not heroic fanfare
- Horror moments: silence before sound, not loud stingers
- Psi elements: low-frequency resonance, not magical shimmer

---

## Writing Style

### Voice
- Direct. Practical. The universe does not explain itself.
- Characters are competent. They speak like people who have done the work.
- No technobabble for its own sake. If it sounds like flavor, make it earn it.
- Dark when it earns it. Dry when it does not.

### NPC Dialogue Rules
1. No more than 3 lines of exposition in a single dialogue node
2. Speak to what the NPC knows and cares about — not the player's menu
3. Rumors should be plausible, not confirmed
4. Leave things unsaid. Not everything gets explained.

### Contract Descriptions
- One paragraph. State the job, hint at the context, identify the risk.
- Never lie in contract text — the tone is frontier realism, not recruitment fraud.
- The reward is stated clearly. The complications are implied.

### Lore Text
- Written from an in-universe perspective (report, record, published text, etc.)
- No author attribution required in Phase 0
- Each entry should feel like it came from a real document in this world
- Avoid direct exposition of mysteries — show the edge, not the answer

---

## UI/UX Principles

- **Hub menus over real-time navigation** — menus are the hub, Phaser scenes handle encounters
- **Information density is acceptable** — players are expected to read
- **No tutorial pop-ups in combat** — if it needs explanation, improve the design first
- **Persistent state is visible** — player should always know their credits, ship hull, and active contract
- **Error states are honest** — "You don't have enough credits" not "Insufficient funds [ERR_0042]"
